01
No Rigidbody2D on Invader Prefab
PHASE 4 CRITICAL
▶
// THE PROBLEM
The Invader prefab is created with only a BoxCollider2D — no Rigidbody2D. Unity's 2D physics requires at least one of the two colliding objects to have a Rigidbody2D for OnTriggerEnter2D callbacks to fire. Without it, bullets pass straight through invaders.
// THE FIX
Add a Rigidbody2D to the Invader prefab with Body Type set to Kinematic and Gravity Scale 0.
Kinematic means Unity tracks the collider but doesn't apply physics forces to it — perfect for objects you move via script.
02
Missing "Is Trigger" Instructions
PHASE 5 CRITICAL
▶
// THE PROBLEM
The plan mentions "Make both colliders Is Trigger = true on the bullets" in a tiny tip — but never makes it a proper step. It also never mentions the Player or Invader collider settings. For OnTriggerEnter2D to fire, at least one collider in the collision must be a trigger. A beginner would miss this entirely.
// THE FIX
Set Is Trigger = true on all four colliders: Player, Bullet, EnemyBullet, and Invader. This guarantees triggers fire regardless of which object hits which.
If Is Trigger is false on both colliders, Unity uses OnCollisionEnter2D instead — which the scripts don't implement.
03
No EnemyBullet Script Provided
PHASE 4 CRITICAL
▶
// THE PROBLEM
The plan says to create a second bullet prefab called EnemyBullet and to "change Vector2.up to Vector2.down in the Bullet script, or make a separate EnemyBullet script" — then provides neither. If the son reuses Bullet.cs, the OnTriggerEnter2D checks for tag "Invader", so enemy bullets hitting the Player do nothing.
// THE FIX
Created a dedicated EnemyBullet.cs script that moves downward and self-destructs off-screen. Collision with the Player is handled from the PlayerController's side.
✓ EnemyBullet.cs (NEW)
public class EnemyBullet : MonoBehaviour { public float speed = 8f; void Update() { transform.Translate(Vector2.down * speed * Time.deltaTime); if (transform.position.y < -6f) Destroy(gameObject); } }
04
Custom Tags Never Created
PHASE 5 MAJOR
▶
// THE PROBLEM
The scripts use CompareTag("Invader") and CompareTag("EnemyBullet"), but the plan only mentions creating these tags in small tips — never as a dedicated setup step. A beginner following the numbered steps would skip the tips and get tag not found errors at runtime.
// THE FIX
Listed tag creation as an explicit step in COLLISION_SETUP.md with a clear table mapping tags to prefabs. Tags must be created via Edit → Project Settings → Tags and Layers before assigning them.
05
GameManager Crashes Before Phase 6
PHASE 5 CRITICAL
▶
// THE PROBLEM
The GameManager (Phase 5) calls UIManager.instance.UpdateScore() and UIManager.instance.UpdateLives() — but UIManager isn't created until Phase 6. Testing collisions at Phase 5 causes a NullReferenceException every time a bullet hits something.
✗ ORIGINAL (crashes)
public void AddScore(int points) { score += points; UIManager.instance.UpdateScore(score); // NULL! }
✓ FIXED (null-safe)
public void AddScore(int points) { score += points; if (UIManager.instance != null) UIManager.instance.UpdateScore(score); }
06
Invader Grid Edge Jitter
PHASE 4 MAJOR
▶
// THE PROBLEM
When an invader hits the screen edge, the grid flips direction and drops. But on the next frame, that invader is still past the edge (the grid only moved down, not back inward), so direction flips again. This creates rapid oscillation — the invaders vibrate at the edge instead of smoothly reversing.
✗ ORIGINAL (jitters)
// Check if any invader has hit the edge foreach (Transform child in transform) { if (child.position.x > edgeLimit || child.position.x < -edgeLimit) { direction *= -1f; // Flips EVERY FRAME transform.Translate(Vector2.down * dropAmount); break; } }
✓ FIXED (cooldown)
if (edgeCooldown > 0f) { edgeCooldown -= Time.deltaTime; } else { foreach (Transform child in transform) { if (child.position.x > edgeLimit || child.position.x < -edgeLimit) { direction *= -1f; transform.Translate(Vector2.down * dropAmount); edgeCooldown = 0.25f; // Don't flip again for 0.25s break; } } }
07
No Win Condition
PHASE 5 MAJOR
▶
// THE PROBLEM
The plan never checks whether all invaders have been destroyed. Once the last invader is killed, the InvaderGrid has zero children but nothing happens — no victory screen, no message, no pause. The player is left in an empty, silent scene with no feedback.
// THE FIX
Added a Victory() method to GameManager and a VictoryPanel to UIManager. InvaderGrid now checks transform.childCount == 0 and triggers the victory flow.
✓ InvaderGrid.cs — win check
if (transform.childCount == 0) { gameEnded = true; if (GameManager.instance != null) GameManager.instance.Victory(); return; }
08
No Invasion Lose Condition
PHASE 4 MAJOR
▶
// THE PROBLEM
In classic Space Invaders, if the alien grid reaches the player's level, the game is over. The plan has no check for this — invaders can march right past the player and off the bottom of the screen with no consequence.
// THE FIX
Added a loseHeight threshold (default Y = -3) in InvaderGrid. When any invader drops below this level, it triggers GameManager.instance.GameOver().
✓ InvaderGrid.cs — invasion check
if (child.position.y < loseHeight) { gameEnded = true; if (GameManager.instance != null) GameManager.instance.GameOver(); return; }
